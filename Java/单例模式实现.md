# Java 单例模式实现

日期：2026-04-19
标签：Java, 设计模式, 单例, 线程安全

## 问题
如何用 Java 实现线程安全的单例模式？

## 回答

### 一、饿汉式（最简单）

```java
public class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```

**优点**：类加载时就创建，天然线程安全
**缺点**：不管用不用都先创建，浪费资源

---

### 二、懒汉式（同步方法）

```java
public class Singleton {
    private static Singleton instance;
    
    private Singleton() {}
    
    public static synchronized Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```

**优点**：延迟加载
**缺点**：每次调用都加锁，性能差

---

### 三、双重检查锁定（DCL）⭐ 推荐

```java
public class Singleton {
    private static volatile Singleton instance;
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        if (instance == null) {              // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {      // 第二次检查（加锁后）
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**为什么需要 volatile？**

防止指令重排序。`new Singleton()` 分三步：
1. 分配内存空间
2. 初始化对象
3. 将引用指向内存

不加 volatile 可能发生 1→3→2，其他线程看到非 null 但未初始化的对象。

---

### 四、静态内部类（优雅）⭐ 推荐

```java
public class Singleton {
    private Singleton() {}
    
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }
    
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

**原理**：类加载机制保证线程安全，调用 `getInstance()` 时才加载 `Holder` 类。

---

### 五、枚举（最佳实践）⭐⭐

```java
public enum Singleton {
    INSTANCE;
    
    public void doSomething() {}
}
```

**优点**：
- 天然线程安全
- 防止反射破坏单例
- 防止序列化破坏单例

---

### 六、对比总结

| 方式 | 线程安全 | 延迟加载 | 性能 | 防反射/序列化 |
|------|---------|---------|------|-------------|
| 饿汉式 | ✓ | ✗ | 高 | ✗ |
| 懒汉式同步 | ✓ | ✓ | 低 | ✗ |
| DCL | ✓ | ✓ | 高 | ✗ |
| 静态内部类 | ✓ | ✓ | 高 | ✗ |
| 枚举 | ✓ | ✗ | 高 | ✓ |

## 相关问题

### 为什么 DCL 需要两次检查？
→ 第一次检查避免不必要的锁竞争，第二次检查防止多个线程同时通过第一次检查后重复创建。

### 静态内部类为什么是线程安全的？
→ JVM 保证类加载过程是线程安全的，只有调用 `getInstance()` 时才触发 `Holder` 类加载。

## 技术拓展

### 反射破坏单例

```java
// 普通单例可被反射破坏
Constructor<Singleton> c = Singleton.class.getDeclaredConstructor();
c.setAccessible(true);
Singleton broken = c.newInstance();  // 创建了新实例！
```

**枚举为什么能防反射？**

枚举的构造方法是私有的，且 JVM 在编译时特殊处理，反射调用枚举构造方法会抛异常。

---

### 序列化破坏单例

```java
// 普通单例序列化后再反序列化会产生新实例
Singleton s1 = Singleton.getInstance();
// 序列化写入文件
ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream("s.obj"));
out.writeObject(s1);
// 反序列化
ObjectInputStream in = new ObjectInputStream(new FileInputStream("s.obj"));
Singleton s2 = (Singleton) in.readObject();
// s1 != s2！破坏了单例
```

**解决方式**：

```java
// 方式一：实现 readResolve 方法
public class Singleton implements Serializable {
    private static final Singleton INSTANCE = new Singleton();
    
    protected Object readResolve() {
        return INSTANCE;  // 返回已有实例，不创建新对象
    }
}

// 方式二：使用枚举（天然支持）
```

### 推荐使用场景

| 场景 | 推荐方式 |
|------|---------|
| 确定会使用 | 饿汉式 |
| 不确定是否使用 | 静态内部类 |
| 需要传参初始化 | DCL |
| 防止反射/序列化攻击 | 枚举 |

### 实际开发中的使用情况

#### Spring Bean 默认就是单例

实际项目中，90%+ 的场景直接用 Spring 管理：

```java
@Service  // 默认就是单例
public class UserService {
    // 不需要自己写单例逻辑
}
```

#### DCL 为什么用得少了？

- 代码复杂，容易写错（忘记 volatile 就出问题）
- JVM 优化后，静态内部类性能足够好
- 有更简洁的替代方案

#### 枚举单例的限制

```java
public enum Singleton {
    INSTANCE;
    
    // 不能继承其他类（enum 默认继承 Enum）
    // 不能有参数化构造
}
```

枚举适合**无状态工具类**，不适合需要复杂初始化的场景。

---

### Spring 单例的坑

**Spring 单例 ≠ 无状态**

不要在单例 Bean 里写成员变量，所有请求会共享：

```java
// 错误：成员变量被所有请求共享
@Service
public class UserService {
    private User currentUser;  // 所有用户共享！危险！
}

// 正确：无状态，只依赖其他 Bean
@Service
public class UserService {
    @Autowired
    private UserRepository repo;  // 依赖注入，安全
    
    public User getUser(Long id) {
        return repo.findById(id);  // 每次调用独立
    }
}
```

---

### 实际选择建议

| 场景 | 推荐方案 |
|------|---------|
| Spring 项目 | `@Service/@Component`（保持无状态） |
| 纯 Java 无状态 | 枚举 |
| 纯 Java 普通 | 静态内部类 |
| 需要传参初始化 | DCL 或用依赖注入框架 |

**核心原则**：能不手写就不手写，用框架管理。
