# 关系型数据库基础知识

日期：2026-04-18
标签：数据库, 事务, 索引, 锁, 隔离级别

## 问题
关系型数据库的核心基础知识有哪些？

## 回答

### 一、事务（Transaction）

事务是一组操作单元，要么全部成功，要么全部失败。核心特性 **ACID**：

| 特性 | 含义 | 说明 |
|------|------|------|
| **A**tomicity（原子性） | 事务是不可分割的工作单位 | 要么全做，要么全不做 |
| **C**onsistency（一致性） | 事务前后数据库状态一致 | 满足所有约束规则 |
| **I**solation（隔离性） | 多个事务互不干扰 | 通过隔离级别控制 |
| **D**urability（持久性） | 事务完成后永久保存 | 即使系统崩溃也不丢失 |

---

### 二、隔离级别（Isolation Level）

从低到高四个级别，解决不同的并发问题：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|---------|------|-----------|------|---------|
| **读未提交（READ UNCOMMITTED）** | 可能 | 可能 | 可能 | 无锁 |
| **读已提交（READ COMMITTED）** | 不可能 | 可能 | 可能 | MVCC/快照 |
| **可重复读（REPEATABLE READ）** | 不可能 | 不可能 | 可能 | MVCC + 间隙锁 |
| **串行化（SERIALIZABLE）** | 不可能 | 不可能 | 不可能 | 全表锁 |

**并发问题解释**：
- **脏读**：读到其他事务未提交的数据
- **不可重复读**：同一事务内，两次读取同一行数据结果不同（被其他事务修改）
- **幻读**：同一事务内，两次查询记录数不同（被其他事务插入/删除）

> MySQL InnoDB 默认隔离级别是 **可重复读**，通过 MVCC + Next-Key Lock 已解决幻读问题。

---

### 三、锁机制（Lock）

锁是实现隔离性的核心手段。

#### 按锁类型分类

| 锁类型 | 说明 | 兼容性 |
|-------|------|--------|
| **共享锁（S锁）** | 读锁，允许并发读 | S+S 兼容，S+X 不兼容 |
| **排他锁（X锁）** | 写锁，独占资源 | 与任何锁都不兼容 |

#### 按锁粒度分类（MySQL InnoDB）

| 锁粒度 | 说明 | 性能影响 |
|-------|------|---------|
| **行锁** | 锁定单行记录 | 并发度高，开销大 |
| **间隙锁（Gap Lock）** | 锁定一个范围，不含记录本身 | 解决幻读 |
| **临键锁（Next-Key Lock）** | 行锁 + 间隙锁 | 解决幻读 |

#### 按锁策略分类

| 策略 | 说明 |
|------|------|
| **乐观锁** | 假设无冲突，提交时检查（CAS、版本号） |
| **悲观锁** | 假设有冲突，操作前加锁（SELECT ... FOR UPDATE） |

---

### 四、索引（Index）

索引是提高查询效率的数据结构，类似书的目录。

#### 常见索引类型

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| **B+树索引** | MySQL 默认，叶子节点存数据/指针 | 范围查询、排序 |
| **哈希索引** | O(1) 查询，不支持范围 | 等值查询 |
| **聚簇索引** | 索引和数据存储在一起 | 主键索引 |
| **非聚簇索引** | 索引和数据分开存储，需回表 | 二级索引 |

#### 索引使用原则

- **最左前缀原则**：联合索引从左匹配
- **覆盖索引**：查询字段都在索引中，避免回表
- **索引失效场景**：`OR`、`LIKE '%xx'`、函数操作、类型转换、`!=`、`IS NULL`

---

### 五、MVCC（多版本并发控制）

MVCC 是 InnoDB 实现高并发读的核心机制，通过**版本链**实现无锁读。

#### 核心组件

1. **隐藏字段**：每行数据包含 `DB_TRX_ID`（事务ID）、`DB_ROLL_PTR`（回滚指针）
2. **Undo Log**：存储数据的历史版本，形成版本链
3. **Read View**：决定当前事务能看到哪个版本

#### 工作原理

- **读操作**：根据 Read View 找到可见版本，不加锁
- **写操作**：修改数据并写入 Undo Log，形成新版本

> MVCC 只在读已提交和可重复读两个隔离级别下生效。

---

## 相关问题

### 为什么 MySQL 选择 B+树而不是 B树？
→ B+树叶子节点形成链表，更适合范围查询；非叶子节点只存索引，一页能存更多键，树更矮，IO 更少。

### 什么是回表？
→ 非聚簇索引存储的是主键值，需要二次查询主键索引获取完整数据，这个过程叫回表。

### 如何避免回表？
→ 使用覆盖索引：查询字段都在联合索引中，直接从索引获取数据。

## 技术拓展

### 三大事务日志详解

#### 一、Redo Log（重做日志）

**定义**：记录"做了什么修改"，用于崩溃恢复。

**本质**：物理日志，记录数据页的具体修改位置和内容。

```
示例：把age 从 10 改成 20
Redo Log记录："第5页第100行的 age 字段改成 20"
```

**存储方式**：固定大小、循环写入

```
┌──────────────────────────────────────────────────┐
│ 已刷盘可覆盖 │空闲区域 │ 已写入未刷盘 │
└──────────────────────────────────────────────────┘
     ↑ checkpoint                    ↑ write pos

write pos → 当前写入位置
checkpoint → 已刷盘位置（可覆盖）
```

**为什么循环写**：
- Redo Log 只用于短期崩溃恢复
-脏页刷盘后，对应日志可覆盖
- 固定空间，不会无限增长

---

#### 二、Undo Log（回滚日志）

**定义**：记录"修改前的值"，用于回滚和 MVCC。

**本质**：逻辑日志，记录反向操作。

```
示例：把age 从 10 改成 20
Undo Log记录："age原值是 10"
```

**用途**：
1. 回滚：事务失败时恢复原值
2. MVCC：构建历史版本，供其他事务读取

**持久化方式**：Undo Log本身也通过 Redo Log 保证持久化

---

#### 三、Binlog（二进制日志）

**定义**：记录执行的 SQL语句或行变更，用于主从复制和备份。

**本质**：逻辑日志，Server 层生成。

```
示例：把 age 从 10 改成 20
Binlog 记录：UPDATE user SET age = 20 WHERE id = 1
或 ROW 格式：id=1 的 age 字段：10 → 20
```

**写入方式**：追加写，不覆盖

```
binlog.000001 → binlog.000002 → binlog.000003 → ...（无限增长）
```

**为什么追加写**：
- 用于长期备份恢复
- 主从同步需要完整历史
- 不会覆盖历史记录

---

#### 四、三大日志对比

| | Redo Log | Undo Log | Binlog |
|--|---------|---------|--------|
| **内容** | 物理日志（页修改） | 逻辑日志（反向操作） | 逻辑日志（SQL/行变更） |
| **归属** | InnoDB 引擎 | InnoDB 引擎 | MySQL Server 层 |
| **写入方式** | 循环写（固定空间） | 持续增长 |追加写（无限增长） |
| **核心用途** | 崩溃恢复（短期） | 回滚、MVCC | 主从复制、备份（长期） |

**设计本质**：
- Binlog 是Server 层通用日志，所有存储引擎共用
- Redo/Undo Log 是 InnoDB 专属，保证引擎层事务特性

---

### 一条 SQL 的完整执行流程

以 `UPDATE user SET age = 20 WHERE id = 1` 为例：

#### 第一阶段：Server 层（接收与准备）

```
① 连接器 → 验证身份、权限，建立连接
② 分析器 → 词法分析：拆出 UPDATE、user、age、id=1
           → 语法分析：检查语法结构是否正确
③ 预处理器 → 检查表/列是否存在、权限是否足够
④ 优化器 → 决定执行方案：用哪个索引
⑤ 执行器 → 准备调用存储引擎
```

#### 第二阶段：InnoDB 引擎层（执行与记录）

```
⑥ 从 Buffer Pool 读取数据页（没有就从磁盘加载）
⑦ 写 Undo Log → 记录 age原值 = 10（用于回滚和MVCC）
⑧ 修改内存数据页 → Buffer Pool 中 age改成20
⑨ 写 Redo Log → redo log buffer（记录修改内容）
⑩ 写Binlog → binlog cache
⑪ 两阶段提交：
    → Redo Log刷盘（prepare 状态）
    → Binlog 刷盘
    → Redo Log刷盘（commit 状态）
⑫ 返回"成功"给客户端
```

#### 第三阶段：异步刷盘（后台线程）

```
⑬ 后台线程异步刷脏页 → 内存修改写入磁盘数据文件
⑭ 推进Checkpoint → Redo Log 对应区域可被覆盖
```

#### 核心要点

| 时机 | 动作 | 目的 |
|------|------|------|
| 事务提交前 | Undo Log 持久化 | 回滚保障 |
| 事务提交时 | Redo Log 持久化 | 崩溃恢复保障 |
| 事务提交时 | Binlog 持久化 | 主从复制、备份 |
| 事务提交后 | 异步刷脏页 | 数据最终持久化 |

---

### 两阶段提交

**目的**：保证 Redo Log 和 Binlog 一致性。

#### 详细流程

```
① Redo Log写入 redo log buffer
② Redo Log刷盘 → 标记为 prepare 状态
③ Binlog 写入 binlog cache
④ Binlog 刷盘
⑤ Redo Log刷盘 → 标记为 commit 状态
⑥ 返回"成功"给客户端
```

#### Prepare 状态含义

"这条 Redo Log已经写好了，但还没最终确认。等Binlog 也写好，才能提交。"

#### 崩溃恢复逻辑

| 崩溃时机 | Redo Log 状态 | Binlog 状态 |恢复动作 |
|---------|--------------|-------------|---------|
|②之后，③之前 | prepare | 无 | 回滚事务 |
|④之后，⑤之前 | prepare | 有（已刷盘） | 提交事务 |
|⑤之后 | commit | 有 | 已成功，无需恢复 |

**核心判断**：
- Redo Log 是 commit → 直接恢复
- Redo Log 是 prepare + Binlog 有记录 →补提交
- Redo Log 是 prepare + Binlog 无记录 → 回滚

**一句话总结**：Redo Log 先"预订"，等 Binlog 确认后再"正式成交"。要么两边都有，要么两边都没有。

---

### WAL（Write Ahead Logging）

**核心原则**：日志先持久化到磁盘，数据文件可以慢慢写。

**性能原理**：
- 写 Redo Log → 顺序 IO，速度快
- 刷数据页 → 随机 IO，速度慢

**设计思路**：把"慢的随机 IO"推迟做，先做"快的顺序 IO"。

### 索引设计原则

#### 索引的代价

| 代价 | 说明 |
|------|------|
| **空间代价** | 索引占用额外存储空间 |
| **时间代价** |每次写入都要维护索引（INSERT/UPDATE/DELETE） |

**设计权衡**：查询变快，写入变慢。索引不是越多越好。

#### 适合建索引的场景

| 场景 | 是否适合 |
|------|---------|
| WHERE 条件频繁出现的字段 | 适合 |
| ORDER BY / GROUP BY 的字段 |适合 |
| JOIN 关联的字段 | 适合 |
| 区分度低的字段（性别、状态） | 不适合 |
| 频繁更新的字段 | 不适合 |

**区分度原则**：字段值越多样，索引效果越好。区分度低于30%不建议建索引。

#### 联合索引与最左前缀原则

联合索引 `KEY idx_a_b_c (a, b, c)`：先按a排序，a相同再按 b排序。

| 查询条件 | 能用索引？ |
|---------|----------|
| `WHERE a = 1` | 能 |
| `WHERE a = 1 AND b = 2` | 能 |
| `WHERE b = 2` | 不能（跳过a） |
| `WHERE a = 1 AND b > 2 AND c = 3` | a、b有效，c失效 |

#### 覆盖索引

查询字段都在索引中，不需要回表，性能提升明显。

```sql
KEY idx_name_age (name, age)

SELECT name, age FROM user WHERE name = 'Alice';  -- 覆盖索引，不回表
SELECT * FROM user WHERE name = 'Alice';          -- 需回表
```

#### 索引下推（ICP）

MySQL 5.6优化：在索引内部就过滤条件，减少回表次数。

### 索引失效场景

| 场景 | 失效原因 |
|------|---------|
| 跳过最左列 | B+树无法定位起始位置 |
| 中间范围查询 | 后续字段无序 |
| LIKE '%xxx' | 无法确定起始位置 |
| 函数操作 | 破坏有序性 |
| 类型转换 | 隐式函数操作 |
| OR单边索引 | 无法单独用索引 |
| != / NOT IN | 无法做范围扫描 |

**排查方法**：用 `EXPLAIN` 查看执行计划，关注 `type`、`key`、`rows`字段。

### SQL 易错场景

#### NULL 值陷阱

```sql
-- NULL 和任何值比较结果都是 NULL，不是 true/false
SELECT * FROM user WHERE age != 20;  -- 不返回 age 为 NULL 的记录

-- 正确写法
SELECT * FROM user WHERE age != 20 OR age IS NULL;
```

#### GROUP BY 规则

```sql
-- 错误：SELECT了非分组列
SELECT city, name, COUNT(*) FROM user GROUP BY city;  -- name不确定！

-- 正确：SELECT的列要么在 GROUP BY 中，要么是聚合函数
SELECT city, MAX(age), COUNT(*) FROM user GROUP BY city;
```

#### WHERE 和HAVING 区别

| | WHERE | HAVING |
|--|-------|--------|
| 执行时机 | 分组前 | 分组后 |
| 能用聚合函数 | 不能 | 能 |

```sql
-- WHERE过滤原始数据
SELECT city, COUNT(*) FROM user WHERE age > 20 GROUP BY city;

-- HAVING过滤分组结果
SELECT city, COUNT(*) FROM user GROUP BY city HAVING COUNT(*) > 10;
```

#### 分页深分页问题

```sql
-- 问题：offset很大时性能下降
SELECT * FROM user LIMIT 100000, 10;

-- 优化：用索引覆盖
SELECT * FROM user 
WHERE id >= (SELECT id FROM user ORDER BY id LIMIT 100000, 1)
LIMIT 10;
```

#### NOT IN 和 NULL

```sql
--危险：子查询返回NULL，NOT IN整体结果为空
SELECT * FROM user WHERE id NOT IN (SELECT user_id FROM order WHERE status = NULL);

-- 安全写法：用NOT EXISTS
SELECT * FROM user WHERE NOT EXISTS (
    SELECT 1 FROM order WHERE order.user_id = user.id
);
```

#### UPDATE忘记WHERE

```sql
-- 灾难：更新全表
UPDATE user SET age = 20;  -- 所有用户都变成20！

-- 安全习惯：先查再改
SELECT * FROM user WHERE city = '北京';
UPDATE user SET age = 20 WHERE city = '北京';
```

### JOIN 用法

#### 四种JOIN类型

| JOIN类型 | 说明 |
|---------|------|
| **INNER JOIN** | 只返回两表都有匹配的记录 |
| **LEFT JOIN** | 左表全部保留，右表无匹配显示 NULL |
| **RIGHT JOIN** | 右表全部保留，左表无匹配显示 NULL |
| **FULL JOIN** | 两表全部保留（MySQL用 UNION模拟） |

#### 常见场景

```sql
-- 找"有订单的用户"
SELECT user.name FROM user INNER JOIN order ON user.id = order.user_id;

-- 找"没有订单的用户"
SELECT user.name FROM user LEFT JOIN order ON user.id = order.user_id
WHERE order.id IS NULL;

-- 多表JOIN
SELECT user.name, order.amount, product.name
FROM user
JOIN order ON user.id = order.user_id
JOIN product ON order.product_id = product.id;
```

#### 易错点

```sql
--错误：LEFT JOIN后用WHERE过滤右表，变成INNER JOIN
SELECT * FROM user LEFT JOIN order ON user.id = order.user_id
WHERE order.amount > 100;  -- 左表无匹配的NULL被过滤

-- 正确：用ON条件
SELECT * FROM user LEFT JOIN order ON user.id = order.user_id AND order.amount > 100;
```
