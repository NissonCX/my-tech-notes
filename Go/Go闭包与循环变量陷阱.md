# Go 闭包与循环变量陷阱

日期：2026-04-21
标签：Go, 闭包, goroutine, 循环变量, 陷阱

## 问题

为什么在 for 循环里启动 goroutine，用循环变量 `i` 会出问题？为什么要用 `go func(n int) {}(i)` 这种写法？

## 回答

### 问题现象

```go
for i := 0; i < 1000; i++ {
    go func() {
        fmt.Println(i)
    }()
}
```

**预期**：打印 0, 1, 2, 3...999

**实际**：可能打印 1000, 1000, 1000... 或其他重复数字

---

### 本质原因：闭包捕获变量本身

#### 循环变量只有一个

`for i := 0; i < 1000; i++` 的执行过程：

```
i = 0 → 检查 → 循环体 → i++ → i 变成 1
i = 1 → 检查 → 循环体 → i++ → i 变成 2
...
i = 999 → 检查 → 循环体 → i++ → i 变成 1000
i = 1000 → 检查不成立 → 循环结束
```

整个循环过程中，**`i` 是同一个变量**，只是值在不断变化。

#### 闭包的本质

`func() { fmt.Println(i) }` 是一个**闭包**。

**闭包是什么？** 一个函数"记住"了它外部定义的变量。

```go
i := 10

f := func() {
    fmt.Println(i)  // 这个函数"记住"了 i，不是"记住" i 的值 10
}

i = 20
f()  // 打印 20，不是 10
```

**关键**：闭包捕获的是**变量本身**，不是变量的**值**。

类比：
- 普通函数参数 = 复制一份（photocopy）
- 闭包捕获变量 = 建立一个引用（指向原件）

---

### 执行过程分析

```go
for i := 0; i < 1000; i++ {
    go func() {
        fmt.Println(i)
    }()
}
```

时间线：

```
时刻 1：i = 0
    → 启动 goroutine A（闭包捕获了 i）
    → goroutine A 还没执行（goroutine 是异步的）

时刻 2：i = 1
    → 启动 goroutine B（闭包也捕获了同一个 i）
    → goroutine A、B 都还没执行

时刻 3：i = 2
    → 启动 goroutine C（闭包也捕获了同一个 i）
    → goroutine A、B、C 都还没执行

...（循环飞快跑完）

时刻 1000：i = 999
    → 启动 goroutine 999
    → 循环结束

时刻 1001：i = 1000（循环结束，i 的值是 1000）

时刻 1002：goroutine A 开始执行
    → fmt.Println(i)
    → i 是多少？i = 1000！（闭包捕获的是同一个 i）

时刻 1003：goroutine B 开始执行
    → fmt.Println(i)
    → i = 1000！
```

**本质**：循环跑得很快，1000 个 goroutine 还没来得及执行，循环就结束了。等 goroutine 开始执行时，`i` 已经是 1000 了。

---

### 解决方案：传参数

```go
for i := 0; i < 1000; i++ {
    go func(n int) {
        fmt.Println(n)
    }(i)
}
```

拆解：

```go
go func(n int) {    // 定义函数，参数 n
    fmt.Println(n)
}(i)                 // 调用函数，把 i 当参数传入
```

**关键**：`(i)` 在循环的**当前时刻**就把 `i` 的值复制给 `n`。

时间线：

```
时刻 1：i = 0
    → 调用 func(0)，把 0 复制给 n
    → goroutine A 启动，n = 0（n 是 goroutine A 自己的变量）

时刻 2：i = 1
    → 调用 func(1)，把 1 复制给 n
    → goroutine B 启动，n = 1（n 是 goroutine B 自己的变量）

...循环继续...

时刻 1000：i = 999
    → goroutine 999 启动，n = 999

时刻 1001：循环结束，i = 1000

时刻 1002：goroutine A 执行
    → fmt.Println(n)
    → n = 0！（n 是 goroutine A 的私有变量，不受 i 影响）
```

**本质**：参数传递是**值复制**，每个 goroutine 有自己独立的 `n`。

---

### 用类比理解

**闭包捕获变量**：
```
100 个人都指着同一个黑板
黑板上的数字在不断变化：0 → 1 → 2 → ...
当每个人开始抄写黑板时，黑板已经变成 1000 了
所有人都抄到 1000
```

**传参数**：
```
老师喊：当前数字是多少？
黑板显示 0 → 老师给第 1 个人一张纸条，写上 0（这张纸条属于第 1 个人）
黑板显示 1 → 考师给第 2 个人一张纸条，写上 1（这张纸条属于第 2 个人）
...
每个人看自己的纸条，纸条不会变
```

---

### 核心区别

| 方式 | 捕获的是什么 | 谁拥有这个变量 |
|------|-------------|---------------|
| 闭包 `func() { println(i) }` | 变量 `i` 本身（引用） | 所有 goroutine 共享同一个 |
| 参数 `func(n int) { println(n) }(i)` | `i` 的值（复制） | 每个 goroutine 有自己的 `n` |

---

### 为什么 Go 这样设计？

**设计哲学**：闭包捕获变量本身，而不是值，是为了灵活性。

例子：
```go
counter := 0

increment := func() {
    counter++  // 闭包可以修改外部变量
}

increment()
increment()
fmt.Println(counter)  // 输出 2
```

如果闭包只捕获值，上面的代码就没法工作了。

**tradeoff**：
- 好处：闭包可以修改外部变量，灵活
- 问题：循环变量共享，容易出错

---

### Go 1.22 的改进

Go 1.22（2024年）改了这个行为——现在 `for` 循环的变量每次迭代都会创建新的。

但在 Go 1.22 之前，这是个经典陷阱，必须用传参数的方式解决。

---

## 一句话总结

**问题的本质**：闭包捕获变量本身（引用），循环只有一个 `i`，所有 goroutine 共享，等执行时 `i` 已经变了。

**解决方案的本质**：传参数是值复制，每个 goroutine 有独立的副本，不受循环变量变化影响。

## 相关问题

### JavaScript 有同样的问题吗？
→ 有。JavaScript 的闭包也是捕获变量本身。但 JavaScript 没有 goroutine，问题不那么明显。

### 为什么不用 `go func(i int) {}(i)` 把参数也叫 `i`？
→ 可以，但容易混淆。用不同的名字（如 `n`）更清晰：`go func(n int) {}(i)`。

### 如果闭包要修改外部变量怎么办？
→ 那就用闭包捕获，但要注意同步问题（用 channel 或锁）。
