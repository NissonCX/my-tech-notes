# Git 数据模型基础概念

日期：2026-04-17
标签：Git, 版本控制

## 问题

Git 数据模型中"快照"是什么意思？为什么不存储差异？这样不会造成资源浪费吗？

## 回答

### 快照 vs 差异

**传统版本控制（如 SVN）**：记录"变化"
- 存补丁文件：第2行从 "World" 改成 "Git"
- 只存改动部分

**Git**：记录"完整快照"
- 存整个文件当前状态
- 不关心改动细节，只关心"现在是什么样子"

### 为什么选择快照？

1. **速度快** — 直接读取，无需逐层应用补丁
2. **分支轻松** — 创建分支只是指向某个快照
3. **安全可靠** — 每个版本独立完整

### 如何避免资源浪费？

Git 通过三种机制节省空间：

**1. 相同文件只存一次**
每个文件内容计算 SHA-1 哈希值，相同内容共享同一个 blob：
```
提交A ──┐
        ├──→ blob(Hello)  ← 只存一份
提交B ──┘
```

**2. 压缩存储**
使用 zlib 压缩算法，文本文件压缩率高。

**3. 打包机制（Pack Files）**
`git gc` 自动将松散对象打包：
- 日常提交：快照存储（简单快速）
- 长期存储：打包后转为差异存储（节省空间）

## 相关问题

- SHA-1 是什么？就是一种哈希算法吗？
- SVN 是什么？

### SHA-1 是什么？

**是的，就是一种哈希算法。**

哈希算法：把任意内容变成固定长度的"指纹"。

```
输入：任意内容（文件、字符串、甚至整个电影）
输出：固定40个字符的十六进制串

例子：
"Hello"  → aaf4c61ddcc5e8a2dabede0f3b482cd9aea9434d
"Hello!" → 03dd9c79d8b3c7e05f8b2e68b0b7e6c14f0e8b3c
```

**关键特性**：

| 特性 | 说明 |
|------|------|
| 唯一性 | 不同输入 → 不同输出（极大概率） |
| 不可逆 | 无法从哈希值还原原始内容 |
| 确定性 | 同样输入 → 永远同样输出 |

**Git 为什么用它？**
- 给每个文件内容生成唯一 ID
- 相同内容 = 相同哈希 = Git 知道"这是同一个文件"
- 校验完整性：内容被篡改，哈希值就变了

### SVN 是什么？

**SVN（Subversion）** 是 Git 之前最流行的版本控制系统，诞生于 2000 年。

| 对比 | SVN | Git |
|------|-----|-----|
| 架构 | 集中式（只有一个服务器） | 分布式（每个人都有完整仓库） |
| 存储 | 差异（存补丁） | 快照（存完整文件） |
| 分支 | 创建分支要复制整个目录 | 创建分支只是一个指针 |
| 离线 | 必联网才能提交、查看历史 | 完全离线也能工作 |

现在 SVN 主要用于传统企业项目，新项目大多用 Git。

## Git 的三种内部对象

Git 仓库内部一切皆为对象，有三种类型：

| 对象 | 存什么 | 类比 |
|------|--------|------|
| blob | 文件内容 | 一张纸（纯内容，没有文件名） |
| tree | 目录结构 | 一个文件夹（记录文件名和子目录） |
| commit | 提交信息 | 一张标签（记录作者、时间、父提交） |

### Blob（文件内容）

最底层的对象，只存文件内容，不存文件名：

```
文件内容：Hello World → blob 对象（SHA-1 = a582b...）
```

**特点**：
- 没有文件名（文件名由 tree 管理）
- 没有路径信息
- 文件改名/移动，内容不变 → blob 不变

### Tree（目录结构）

管理文件名和目录层级，相当于"文件夹"：

```
tree 对象
├── blob(a582b...) → readme.md
├── blob(c3d4e...) → config.json
└── tree(f5g6h...) → src/
    └── blob(x7y8z...) → main.py
```

### Commit（提交信息）

记录"谁、何时、为什么"：

```
commit 对象
├── tree: 指向某个 tree
├── parent: 指向上一 commit
├── author: 作者 + 时间
├── message: 提交说明
```

## 有向无环图（DAG）

Git 历史就是一个 DAG：

| 词 | 含义 |
|------|------|
| 有向 | 箭头有方向（commit → parent） |
| 无环 | 没有回路（commit 创建后 parent 固定，无法形成回路） |
| 图 | 节点(commit) + 连线(parent指向) |

### 分支和合并

**分支**：在 DAG 上开一条新线
```
      C4 ← C3 ← C2 ← C1  ← main
     /
C5 ←                    ← feature
```

**合并**：两条线汇合
```
      C6 ← merge commit（parent: C4 + C5）
     /  \
    C4   C5
     \  /
      C3 ← C2 ← C1
```

### DAG 的好处

- 有向：历史有明确时间顺序
- 无环：永远能追溯到根提交
- 多父节点：合并操作自然表示

## 快照结构示例分析

```
<root> (tree)
│
├── foo (tree)
│   └── bar.txt (blob, 内容 = "hello world")
│
└── baz.txt (blob, 内容 = "git is wonderful")
```

**存储形式**：

```
tree <root>:
  100644 blob 4448adbf...    baz.txt
  040000 tree c68d233a...    foo
```

| 模式 | 类型 | SHA-1 | 文件名 |
|------|------|-------|--------|
| 100644 | blob | 4448adbf... | baz.txt |
| 040000 | tree | c68d233a... | foo |

- `100644`：普通文件权限
- `040000`：目录权限
- tree 记录文件名 + SHA-1 指向
- blob 只存内容，文件名在上层 tree

**关键点**：tree 是"目录表"，blob 是"文件内容"，SHA-1 是"指针"。

## 相关问题

- 大厂开发过程的 Git 分支工作流是怎样的？
- MR 是什么？

### Git Flow 工作流

```
main (生产分支) ← 只存稳定发布版本
  │
  ↓ merge (发布时)
dev (开发分支) ← 集成所有功能
  │
  ↓ merge (功能完成后)
feature/xxx (功能分支) ← 开发具体功能
```

**流程**：
1. 从 dev 创建 feature 分支
2. 在 feature 分支开发、提交
3. 功能完成 → 提 MR/PR → 代码审查
4. 审查通过 → merge 到 dev
5. dev 稳定后 → merge 到 main（发布）

**其他分支**：
- `hotfix/xxx`：紧急修复，从 main 分出
- `release/xxx`：发布准备，从 dev 分出

### Trunk Based 工作流

很多大厂（Google、Meta）采用更简单的方式：
```
main (主干) ← 所有开发都在这里
  ↓ 短期 feature 分支（几小时~几天就合并）
```

**结构对比**：
```
Git Flow:        main → dev → feature (三层)
Trunk Based:     main → feature         (两层，省去 dev)
```

**本质区别**：

| 区别 | Git Flow | Trunk Based |
|------|----------|-------------|
| 分支存活时间 | feature 可能存活数周 | feature 存活几小时~几天 |
| 合并频率 | 功能完成后才合并 | 每天甚至每小时合并 |
| 心态 | "先做完再集成" | "持续集成，随时可发布" |
| main 的状态 | main ≠ 最新代码，dev 才是 | main = 最新代码，随时可发布 |

**核心差异**：
- Git Flow：main 是"发布历史"，dev 是"开发主线"
- Trunk Based：main 就是"开发主线 + 发布历史"，合二为一

**为什么大厂敢这样做？有配套保护措施：**

| 保护措施 | 说明 |
|----------|------|
| 短期分支 | 分支存活时间短，冲突概率低 |
| 强制代码审查 | PR/MR 必须有人审查通过 |
| 自动化测试(CI) | 测试不通过不能合并 |
| Feature Flags | 未完成功能用开关控制，不影响用户 |

**长期分支的问题**：分支开发时间长 → main 变化多 → 合并时大量冲突。

**适用场景**：
- Git Flow：小团队、发布周期长、测试覆盖低
- Trunk Based：大厂、自动化完善、测试覆盖高

### MR vs PR

| 平台 | 名称 |
|------|------|
| GitHub | Pull Request (PR) |
| GitLab | Merge Request (MR) |

**本质相同**：请求把分支合并到目标分支，附带代码审查流程。

## 常用 Git 命令

### 基础操作

| 命令 | 说明 |
|------|------|
| `git init` | 初始化仓库 |
| `git clone <url>` | 克隆远程仓库 |
| `git status` | 查看当前状态（修改、暂存、未跟踪） |
| `git add <file>` | 添加文件到暂存区 |
| `git add .` | 添加所有修改到暂存区 |
| `git commit -m "msg"` | 提交暂存区内容 |
| `git diff` | 查看未暂存的修改 |
| `git diff --staged` | 查看已暂存的修改 |

### 分支操作

| 命令 | 说明 |
|------|------|
| `git branch` | 列出所有本地分支 |
| `git branch <name>` | 创建新分支 |
| `git checkout <branch>` | 切换分支 |
| `git checkout -b <name>` | 创建并切换到新分支 |
| `git switch <branch>` | 切换分支（新版推荐） |
| `git switch -c <name>` | 创建并切换（新版推荐） |
| `git merge <branch>` | 将指定分支合并到当前分支 |
| `git branch -d <name>` | 删除分支 |
| `git branch -D <name>` | 强制删除分支 |

### 远程操作

| 命令 | 说明 |
|------|------|
| `git remote -v` | 查看远程仓库 |
| `git remote add origin <url>` | 添加远程仓库 |
| `git push origin <branch>` | 推送分支到远程 |
| `git push -u origin main` | 推送并设置上游（首次推送） |
| `git pull origin <branch>` | 拉取远程分支并合并 |
| `git fetch` | 获取远程更新（不合并） |

### 历史查看

| 命令 | 说明 |
|------|------|
| `git log` | 查看提交历史 |
| `git log --oneline` | 简洁显示（每行一个提交） |
| `git log -n 5` | 查看最近5条提交 |
| `git log --graph` | 图形化显示分支结构 |
| `git show <commit>` | 查看某次提交详情 |
| `git blame <file>` | 查看文件每行的修改者 |

### 撤销与回退

| 命令 | 说明 |
|------|------|
| `git restore <file>` | 撤销工作区修改（新版） |
| `git restore --staged <file>` | 取消暂存（新版） |
| `git reset HEAD <file>` | 取消暂存（旧版） |
| `git reset --soft HEAD~1` | 回退1次提交，保留修改在暂存区 |
| `git reset --hard HEAD~1` | 回退1次提交，丢弃所有修改 |
| `git revert <commit>` | 创建新提交来撤销指定提交 |

### 暂存工作区

| 命令 | 说明 |
|------|------|
| `git stash` | 暂存当前修改 |
| `git stash list` | 查看暂存列表 |
| `git stash pop` | 恢复最近暂存并删除记录 |
| `git stash apply` | 恢复暂存但保留记录 |

### 其他实用命令

| 命令 | 说明 |
|------|------|
| `git gc` | 垃圾回收，打包压缩对象 |
| `git reflog` | 查看操作历史（可用于恢复误删提交） |
| `git cherry-pick <commit>` | 将某次提交应用到当前分支 |
| `git rebase <branch>` | 变基（将当前分支移植到指定分支上） |
