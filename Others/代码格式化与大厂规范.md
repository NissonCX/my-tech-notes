# 代码格式化与大厂规范

日期：2026-04-18
标签：开发规范, 格式化工具

## 问题

代码格式化是什么？大厂如何使用格式化工具和规范？

## 回答

### 为什么格式化重要？

不同团队成员代码风格不同：
- 格式差异分散代码审查注意力
- 看不出真正改动，只看到格式改动
- 团队争论"哪种风格更好"，浪费时间

**格式化工具的作用**：消除无意义的风格差异。

### 三种主流格式化工具

#### 1. Prettier（多语言）

**特点**：
- 支持 JS、TS、CSS、HTML、Markdown 等
- 高度可配置，推荐"少配置"
- IDE 集成广泛

```bash
npm install --save-dev prettier
npx prettier --write .    # 格式化
npx prettier --check .    # 只检查
```

**大厂实践**：CI 强制检查 + 配合 ESLint

#### 2. Black（Python）

**特点**：
- "不可配置"——消除所有争论空间
- 官方口号："The Uncompromising Code Formatter"
- Dropbox、Mozilla 内部强制使用

```bash
pip install black
black .
```

#### 3. gofmt（Go）

**特点**：
- Go 官方工具，不可配置
- 所有 Go 代码格式完全一致

Go 社区哲学："没有风格争论，因为只有一种风格"

### 格式化 vs Lint 的分工

| 类型 | 职责 | 示例 |
|------|------|------|
| 格式化 | 代码外观 | 引号、空格、缩进、行长度 |
| Lint | 代码质量 | 未使用变量、逻辑错误、安全漏洞 |

```javascript
// 格式化问题（Prettier）
const x=1+2  →  const x = 1 + 2

// Lint 问题（ESLint）
const x = 1  // 变量未使用
```

### 大厂规范实践

**规范层级**：
```
格式化 → 风格统一
    ↓
Lint → 捕获潜在问题
    ↓
自动化 → CI + pre-commit hook 强制执行
```

**大厂做法**：

| 层级 | 工具 | 执行方式 |
|------|------|----------|
| 格式化 | Prettier/Black/gofmt | CI 必须通过 |
| Lint | ESLint/Pylint | CI 必须通过 |
| 自动执行 | pre-commit hook | 提交前自动格式化 |

**pre-commit Hook 示例**：
```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/pre-commit/mirrors-prettier
    hooks:
      - id: prettier
  - repo: https://github.com/psf/black
    hooks:
      - id: black
```

**Google 的"零容忍"策略**：
- 格式化检查失败 → CI 直接拒绝
- 没有"手动格式化"的机会
- 防止 diff 被格式污染

## Linting（代码检查）

### Lint 是什么？

**静态分析**：不运行代码，只读代码就能发现问题。

| 方式 | 需运行？ | 发现什么？ |
|------|---------|-----------|
| 单元测试 | 是 | 功能是否正确 |
| Lint | 否 | 代码结构、潜在 bug、反模式 |

**Lint 能发现的问题**：

```python
# 未使用变量
x = 1  # 定义了但没用

# 反模式
if x == True:  # 应写成 if x:

# 安全隐患
subprocess.Popen(cmd, shell=True)  # 可能被注入
```

### 三种主流 Lint 工具

#### 1. ESLint（JS/TS）

最流行的 JS Lint 工具：

```javascript
// 未使用变量
const unused = 1;  // ESLint 报错

// 反模式
for (var i = 0; i < arr.length; i++) {  // var 应改为 let

// 潜在 bug
if (x = 1) {  // 应是 == 或 ===，不是赋值
```

**大厂实践**：Airbnb、Google 有著名规则集。

#### 2. Ruff（Python）

新一代 Python Lint 工具：
- 极快（比 flake8 快 10-100 倍）
- lint + format 合一（替代 flake8 + Black）
- Rust 实现

```bash
pip install ruff
ruff check .    # Lint
ruff format .   # 格式化
```

#### 3. Semgrep（跨语言语义检查）

**特点**：
- 不只是语法检查，而是语义检查
- 在 AST 层面匹配代码模式
- 跨语言支持

**普通 grep vs Semgrep**：

```bash
# 普通 grep：只匹配文本
grep "shell=True" file.py

# Semgrep：匹配语义等价代码
semgrep -l python -e "subprocess.Popen(..., shell=True)"
# 能找到：
# subprocess.Popen(cmd, shell=True)
# subprocess.Popen(cmd, True, shell=True)  # 参数位置不同也能找到
# x = True; subprocess.Popen(cmd, shell=x)  # 变量赋值也能找到
```

**大厂用途**：安全扫描、代码审计（Dropbox、Shopify）

### 工具分工

| 工具类型 | 检查什么 | 示例 |
|----------|----------|------|
| 格式化 | 代码外观 | Prettier、Black |
| Lint | 代码质量 | ESLint、Ruff |
| 类型检查 | 类型错误 | TypeScript、mypy |
| 语义检查 | 危险代码模式 | Semgrep |

### 大厂 CI 流水线示例

```yaml
jobs:
  lint:
    steps:
      - run: npm run lint       # ESLint
      - run: npm run typecheck  # TypeScript
      - run: semgrep --config=auto  # 安全扫描
```

**流程**：任何检查不通过 → 不能合并

## Testing（测试）

### 测试层级

| 类型 | 范围 | 占比（Google标准） |
|------|------|-------------------|
| 单元测试 | 单个函数 | 70% |
| 集成测试 | 模块间交互 | 20% |
| 端到端测试 | 全流程 | 10% |

**原则**：单元测试量大、快；端到端测试量少、慢。

### 开发者测试做到什么程度？

**核心原则**：追"信心"，不是追覆盖率数字。

**判断标准（问自己 3 个问题）**：

1. 这段代码出 bug 影响多大？→ 影响大必须测
2. 逻辑复杂吗？→ 复杂必须测
3. 会被别人改吗？→ 经常改动需要测试保护

**不同代码的策略**：

| 代码类型 | 测试要求 |
|----------|----------|
| 业务核心逻辑 | 必须单元测试 |
| API 接口 | 必须集成测试 |
| 工具类/算法 | 必须单元测试（边界条件多） |
| 简单 CRUD | 可选 |
| 前端 UI | 可选 |

### 回归测试

修复 bug 后必须写测试防止复发：

```python
def test_empty_username():
    with pytest.raises(ValueError):
        login(username="", password="xxx")
```

### TDD 与 Mocking

**适合 TDD**：算法、复杂逻辑、接口设计
**不适合 TDD**：探索性开发、前端 UI

**Mock 使用**：
- 内部逻辑 → 不用 Mock
- 外部依赖 → 用 Mock 隔离

### Code Coverage 怎么看？

| 覆盖率 | 态度 |
|--------|------|
| < 50% | 可能不够 |
| 50-80% | 合理区间 |
| > 80% | 成本高，收益低 |

**关键**：覆盖率只是参考，测核心逻辑比测 trivial 代码更有价值。

**完成测试的标准**：核心逻辑有测试 + 关键接口有测试 + 修复的 bug 有回归测试

## Pre-commit Hooks

Git 钩子框架，提交前自动运行格式化、Lint、单元测试。

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    hooks:
      - id: ruff
      - id: ruff-format
```

## CI/CD（持续集成/部署）

### 流程图

```
代码提交
    ↓
┌─────────── CI（持续集成）───────────┐
│  格式化 → Lint → 类型检查 → 测试     │
│  任何失败 → 停止，不能合并            │
└─────────────────────────────────────┘
    ↓ 全部通过
┌─────────── CD（持续部署）───────────┐
│  构建 → 部署 → 验证                  │
└─────────────────────────────────────┘
```

### CI vs CD

| 阶段 | 做什么 | 失败后果 |
|------|--------|----------|
| CI | 检查代码质量、跑测试 | 不能合并 |
| CD | 构建、发布到服务器 | 不能上线 |

### 为什么顺序重要？

- 格式化最快 → 先跑
- Lint 也快 → 第二步
- 测试最慢 → 最后跑
- 前面失败就停，节省时间

### Pre-commit vs CI

| 时机 | 运行位置 | 检查内容 |
|------|----------|----------|
| Pre-commit | 本地 | 格式化 + Lint（快速） |
| CI | 服务器 | 格式化 + Lint + 测试（完整） |

**都要**：Pre-commit 快速拦截，CI 最终兜底。

### 部署策略

| 策略 | 说明 |
|------|------|
| 分阶段部署 | dev → staging → prod |
| 蓝绿部署 | 新旧版本并存，验证后切换 |
| 金丝雀发布 | 先部署 5% 用户，逐步放开 |

**一句话**：CI = 代码体检（能否合并），CD = 代码发布（能否上线）
